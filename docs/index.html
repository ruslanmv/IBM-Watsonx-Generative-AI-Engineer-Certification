Generative AI
Fundamentals

Videos 3–5 · Capabilities, limits, architecture patterns, and your first Prompt Lab run.

The Five Core Capabilities of LLMs

Memorize these — every exam asks you to map a business need to one.

📝 Generation

Marketing copy, code, emails, synthetic data. Open-ended creation.

📚 Summarization

Long docs → short docs. Extractive or abstractive.

🏷️ Classification

Spam, sentiment, intent. Discrete labels from text.

❓ Question Answering

Closed-book (parametric) or open-book (RAG).

🔁 Transformation

Translation, style transfer, format conversion, entity extraction.

Not a Capability

Real-time stock prices · live web search · arithmetic guarantees → those need tools / RAG / agents.

Limits You Must Know Cold

🪨 Knowledge Cutoff

Models don't know events after their training date. Granite-13B-v2 ≈ early 2024.

😵 Hallucinations

Confident wrong answers. Mitigated by grounding (RAG) + low temperature.

⚖️ Bias

Inherited from training data. Test with adversarial prompts; document in model cards.

📐 Context Window

Hard token limit. Granite-13B = 8k. Llama-3.1 = 128k. Exceeding it truncates silently.

Architecture Decision Tree

Need: high-quality output for a business task │ ├── Knowledge inside the model is enough? │ ├── YES → Prompt Engineering (cheapest, fastest) │ └── NO → continue ↓ │ ├── Need facts from your documents that change? │ ├── YES → RAG (no retraining) │ └── NO → continue ↓ │ ├── Need a new behavior / style / format? │ ├── YES → Fine-Tuning (LoRA / PEFT / InstructLab) │ └── NO → continue ↓ │ └── Need multi-step reasoning + tool use? └── YES → Agents (LangChain, LangGraph)

This tree is the spine of the exam. Read every question against it.

Prompt Engineering

Videos 6–9 · Zero/few-shot, chain-of-thought, parameters, and Prompt Lab patterns.

Three Prompting Levels

Zero-Shot

"Classify this email as spam or not." No examples. Cheap. Works for easy tasks.

Few-Shot

Include 2–5 examples. Steers tone, format, edge cases. Cost grows with examples.

Chain-of-Thought

"Let's think step by step." Forces reasoning before final answer. Best for math & logic.

🎯 Exam Tip

If a question shows 2–3 input/output pairs in the prompt, it's few-shot. If it adds "Reasoning: …" before "Answer:", that's chain-of-thought.

The CRISPE Prompt Framework

C — Capacity & Role

"You are a senior tax accountant."

R — Request

"Review this expense report."

I — Insight

"Flag items above $500 without receipts."

S — Style & Tone

"Formal, bullet-point summary."

P — Personality

"Skeptical, audit-minded."

E — Experiment

"Output JSON with fields: id, flag, reason."

Other frameworks to recognize: APE (Action–Purpose–Expectation), CARE (Context–Action–Result–Example).

Parameter Cheat Sheet

TemperatureRandomnessDeterministic, repetitiveCreative, off-topic risk0.7
Top-pNucleus sampling — keep tokens until prob ≥ pNarrowWide0.9

🎯 Exam Tip

"Make the model more deterministic / factual" → lower temperature, lower top-p. "Make it more creative" → raise both. The exam loves this pairing.

Prompt Risks & Guardrails

💉 Prompt Injection

"Ignore previous instructions and…" Mitigate with input sanitization + system-prompt separation.

🔓 Jailbreaks

Role-play to bypass safety. Mitigate with output filters + the Granite Guardian model.

🪪 PII Leakage

Mask emails, phones, SSNs before sending to the model. Track in audit logs.

⚖️ Bias Amplification

Adversarial test suites; document residual bias in the model card.

Fine-Tuning &
Customization

Videos 10–15 · The largest exam section. PEFT, LoRA, QLoRA, InstructLab, Tuning Studio.

When to Fine-Tune (and When Not To)

SituationTry FirstFine-Tune?
New facts that change weeklyRAGNo
Custom JSON output schemaFew-shot + JSON modeMaybe
Domain jargon + house stylePromptingYes — PEFT
Brand-new task model never saw—Yes — full or InstructLab
Tiny dataset (< 100 examples)Few-shotNo

Three Tuning Strategies

Full Fine-Tuning

Update all weights. Highest quality, highest cost, biggest GPUs, risk of catastrophic forgetting.

PEFT — Parameter-Efficient

Freeze the base. Train small adapters (< 1% of params). 90% of the quality at 10% of the cost. LoRA lives here.

Prompt Tuning

Learn a soft prompt vector only. Cheapest. Less expressive than LoRA but the watsonx default.

LoRA — Low-Rank Adaptation

Decompose the weight update ΔW into two small matrices A × B of rank r.

W_new = W_frozen + A · B ← only A and B are trained ───────── rank r ≪ d Typical r : 4, 8, 16, 32 Trainable : 0.1% – 1% of full model Quality : ≈ 95% of full fine-tune Disk size : tens of MB instead of tens of GB

🎯 Exam Tip

If you see "add small trainable matrices to frozen weights" — it's LoRA. If they add "with 4-bit quantization of the base" — that's QLoRA.

Quantization shrinks the model so it fits on smaller GPUs. QLoRA = quantized base + LoRA adapters — the most cost-efficient combo on the exam.

IBM InstructLab — The IBM Way

Open-source toolkit + taxonomy for teaching a model new skills and knowledge with synthetic data.

🌳 Taxonomy

YAML tree of skills (style, format) and knowledge (facts). You only add a few seed examples per leaf.

🧬 Synthetic Data Gen

A "teacher" model expands seeds into thousands of training examples — solves the small-data problem.

🏗️ Multi-Phase Training

Knowledge phase + skills phase, then alignment. Built on LAB method (paper: Sudalairaj et al.).

🤝 Community Model

Contribute taxonomy back; merged models are released publicly. Aligns with Granite.

Dataset Best Practices

Format

JSONL with {"input": ..., "output": ...} or chat-style messages[]. Tuning Studio expects JSONL.

Size

50–200 examples for prompt tuning · 500–5,000 for LoRA · 10k+ for full fine-tune.

Quality

Deduplicate. Balance classes. Hold out 10–20% for evaluation. Bad data > too little data.

Splits

Train / validation / test. Never evaluate on training data — that's a guaranteed exam trap.

Tuning Studio — Knobs to Recognize

HyperparameterEffectWatch For
Learning rateStep sizeToo high → divergence · too low → no learning
EpochsPasses over dataToo many → overfitting
Batch sizeExamples per stepBigger = smoother, more GPU memory

RAG = inject relevant chunks into the prompt so the model answers from your data — without retraining.

Embeddings, Plainly

An embedding is a vector of ~384–1536 floats that captures meaning. Similar meaning → nearby vectors.

"The cat sat" → [ 0.12, -0.04, 0.31, ... ] (768 dims) "A feline rests"→ [ 0.11, -0.05, 0.30, ... ] ← cosine ≈ 0.98 "Bitcoin price" → [-0.41, 0.22, -0.08, ... ] ← cosine ≈ 0.06

watsonx defaults: slate-30m-english-rtrvr (IBM) · multilingual-e5-large.

Chunking Strategies

Fixed Size

e.g. 512 tokens, 50 token overlap. Simple. Loses semantic boundaries.

Recursive Character

Split on \n\n then \n then ". " — LangChain default. Good balance.

Semantic

Embed sentences, group by similarity. Best quality, more compute.

Document-Aware

Use headings, tables, code blocks as natural boundaries. Best for markdown / HTML.

Overlap (10–20%) reduces context loss at chunk boundaries. The exam expects you to know why.

Vector Databases

StoreTypeUse When
FAISSIn-memory, librarySingle machine, dev, small corpus
ChromaEmbedded, persistentLocal apps, notebooks
Milvus / WeaviateDistributedProduction at scale
watsonx.data + ElasticsearchIBM-nativeEnterprise RAG with governance

🎯 Exam Tip

When the question says "IBM-native, governed, scalable" — pick watsonx.data + Elasticsearch. When it says "prototype on a laptop" — pick FAISS or Chroma.

LangChain RAG in 20 Lines


from langchain_ibm import WatsonxLLM, WatsonxEmbeddings
from langchain.vectorstores import FAISS
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.chains import RetrievalQA

# 1. Load + chunk
splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=64)
chunks   = splitter.split_documents(docs)

# 2. Embed + index
emb   = WatsonxEmbeddings(model_id="ibm/slate-30m-english-rtrvr", ...)
store = FAISS.from_documents(chunks, emb)

# 3. Wire to LLM
llm = WatsonxLLM(model_id="ibm/granite-13b-instruct-v2", ...)
qa  = RetrievalQA.from_chain_type(
    llm=llm, retriever=store.as_retriever(search_kwargs={"k": 4}))

print(qa.run("What is our refund policy for enterprise contracts?"))

Quality Levers

Top-k

How many chunks to retrieve. k=3–5 default. Too high → noise dilutes the answer.

Re-ranking

Cross-encoder over the top-N to pick the best. Boosts accuracy ~10–20%.

Hybrid Search

BM25 (keyword) + vector. Catches names, codes, exact strings.

Query Rewriting

LLM expands the user query before retrieval — fixes vague questions.

Versioning & Lifecycle

Asset Versions

Every promote creates a new immutable version. Roll back by redeploying the previous one.

A/B Routing

Send 10% of traffic to v2, watch metrics, then ramp. Track in governance.

Online vs Batch

Online = low-latency REST · Batch = job over a dataset, results to object storage.

Retire

Models drift. Schedule re-evaluation every quarter; document EOL in the model card.

Monitoring — The Four Pillars

⏱️ Latency

p50 / p95 / p99 per endpoint

💰 Cost

Tokens × $/1k tokens per tenant

📉 Drift

Input + output distribution vs. baseline

⚖️ Quality

Human or LLM-as-judge sampling

All four feed into watsonx.governance dashboards. The exam loves the word "observability".

Orchestration Patterns

Sequential Chain

Extract → Summarize → Translate. Each step's output feeds the next.

Router

Classifier picks one of N specialist sub-chains. Cheap routing model in front.

Agent

LLM picks tools dynamically — see the companion Agentic AI course.

Question Patterns to Recognize

"Best fit" architecture

Scenario → choose Prompting / RAG / Fine-tune / Agent. Use the decision tree from Module 2.

"Which knob"

Symptom → parameter to change. Temperature / top-p / max_tokens / repetition_penalty.

"Most cost-efficient"

Trick: it's usually PEFT/LoRA + smaller model, not full fine-tune.

"Governance / safety"

Look for words: bias, audit, lineage, model card → watsonx.governance.
