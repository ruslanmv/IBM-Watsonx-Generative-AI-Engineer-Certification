\chapter{Deployment}
\label{ch:deployment}

\epigraph{``Notebooks don't pay invoices --- endpoints do.''}{}

The Acme Assistant worked beautifully in a notebook. It answered policy questions with citations, drafted bank-style emails, summarised long claim threads, and refused to answer anything outside its scope. None of that mattered to the eight hundred customer-support agents who needed to use it, because the notebook ran on the AI engineer's laptop. Deployment day is the day the Assistant becomes a product.

Section~5 of the exam --- thirteen percent of the test --- is the set of decisions that day surfaces. How does a prompt template, a tuned model, or a custom AI service move from a project to a production REST endpoint? Where does governance attach? Which one calls it: an interactive user, a batch job, both? What sits in front of multiple models so the application only has to know one URL? How do you tell whether the deployed thing is still working? This chapter walks the lifecycle in the order an engineer would meet it, with the Acme rollout as the running example. Where the deployed artefact is a tuned model, refer back to Chapter~\ref{ch:fine-tuning}; where it serves a RAG corpus, Chapter~\ref{ch:rag}.

\section{Planning the deployment (Objective 5.1)}

A deployable asset on watsonx --- a prompt template, a tuned model, a custom AI service --- has a lifecycle that always looks the same. It is built in a \term{project}, evaluated against a held-out set, promoted into a \term{Deployment Space}, evaluated again, promoted into staging, evaluated again, promoted into production, monitored, eventually retired. Every promote creates a new immutable version of the asset, and applications call the deployed endpoint by ID, never by reference to the editable project artefact. That single discipline is what makes the audit trail real.

The pattern for managing changes follows from there. Never edit a production prompt in place; copy it, change it, promote it as a new version, A/B test if you can, then swap. Pin the model identifier inside the prompt template so the model itself becomes part of the asset --- changing the model in production is a new version, not a hot edit. Run a small \emph{golden set} of canonical inputs against the candidate version before flipping traffic; the regressions you catch in five minutes here are the ones you would otherwise debug at 3 am. Roll back by re-pointing to the previous immutable version, which is a single API call away.

\subsection{What governance asks for before production}

Before a watsonx deployment is considered production-ready, the governance team needs three things on the asset. \emph{Model performance} --- baseline metrics on a held-out evaluation set (accuracy, F1, ROUGE, or a custom rubric, depending on the task). \emph{Evaluate inferences} --- live evaluators (LLM-as-judge, drift, fairness) attached to the deployment so quality degradation is detected in production, not on the next release. \emph{Explain outcomes from prompts} --- the system can produce an audit-grade explanation that names which prompt template, which model version, which retrieved chunks, which parameters, and which response, for any inference. At Acme, the engineer's first deployment failed governance review because the explanation trail had no record of the retrieved chunks; they added the citation logger and re-promoted.

\examtip{Phrases like \emph{``explain outcomes''}, \emph{``audit-grade trail''}, or \emph{``factsheet''} all route to \textbf{watsonx.governance} sitting on top of a Deployment Space.}

\section{Deploying assets (Objective 5.2)}

\begin{figure}[h]
\centering
\includegraphics[width=0.98\textwidth]{assets/figures/fig20-deployment-promotion}
\caption{The promotion flow: a prompt or model graduates from Project to Dev Space, Staging (where governance attaches the factsheet), and finally Production with a canary rollout. Any stage can roll back.}
\label{fig:promotion-flow}
\end{figure}

A \term{Deployment Space} is the production sibling of a project --- a permission-scoped, audited container for promoted assets and the REST endpoints they expose. Spaces are how watsonx separates \emph{building} from \emph{serving}. Inside a project you have an editable artefact, low ceremony, easy iteration. Inside a space you have an immutable version, governance attached, and a stable endpoint that an application can call.

The benefits stack up the moment a team uses spaces seriously. Lifecycle governance becomes the promotion trail across Dev, Eval, and Prod spaces, which is the audit trail. Each space exposes its own endpoint URLs, so the application points at the right environment without code changes. Access control runs independently per space, which prevents the long-running organisational mistake of over-permissioning ML engineers into production. Assets are immutable versions rather than editable artefacts, so reproducibility comes for free. Per-deployment metrics, logs, and governance hooks make observability per-asset rather than per-namespace.

For prompt templates specifically, the win is sharper. One REST endpoint with the same shape across all environments means the application code does not change between Dev and Prod. The model, parameters, and few-shot examples are all baked into the asset, which makes it impossible to skew the behaviour through stray app-side configuration. And every update requires an explicit promote, which gives security and governance an inspection point on every change.

The mechanical shift in the calling application is small. Before adopting a Deployment Space the app was calling the foundation-model SDK directly with a full prompt string. Afterwards it calls a stable deployment endpoint and passes only the variables.

\begin{lstlisting}[style=pythonstyle]
# Before --- the app holds the prompt
model = ModelInference(model_id="ibm/granite-3-2-8b-instruct", ...)
out   = model.generate_text(big_prompt_string)

# After --- the app passes only variables
out = client.deployments.generate_text(
    deployment_id = TICKET_CLASSIFIER_PROD_ID,
    params={"prompt_variables": {"ticket_text": user_input}})
\end{lstlisting}

The endpoint hides the model identifier, the parameters, the few-shot block, and the stop sequences. The app passes only variables. Cleaner, safer, governed.

\section{Custom models (Objective 5.3)}

A \term{custom model} is a tuned (LoRA, prompt-tuned, or InstructLab) or imported foundation model that does not appear in the off-the-shelf watsonx catalogue. The deployment story is the same as for a catalogue model, with a handful of extra preconditions. The model architecture has to be compatible with watsonx's custom-model serving (the Granite, Llama, and Mistral families today). The licence has to be approved --- open weights, indemnified, or your own. The model files have to be in a supported format (HuggingFace, GGUF, or watsonx tuning artefacts). The watsonx.ai plan tier has to support custom-model serving (a higher tier than catalogue-only). And the GPU profile has to match the model size --- a single L4 for a small 8B, an A100 for 13B, multi-GPU for 70B.

From the application's perspective the custom model is just another asset. It lives in your project or space with its own ID, you wrap it in a prompt template or AI service if you want the same governance posture as a foundation model, and applications call it through the same \texttt{/ml/v1/deployments/\{id\}/text/generation} or \texttt{/text/chat} endpoint they used for the catalogue model. The Acme Assistant's LoRA-tuned email drafter --- the one that survived Chapter~\ref{ch:fine-tuning} --- went through this exact path.

\section{Versioning, gateways, and the architecture decisions (Objective 5.4)}

Every promote of an asset into a space creates a new version. The deployment can point to a specific version, and you can re-point it with a single API call. Rollback is just \emph{re-deploy version N--1}.

If a new prompt-template version preserves its input variables and output schema, the calling application does not change at all --- the same deployment ID with the same variable contract now behaves differently. That is the entire point of variables and schemas. If the schema changes, you should create a new deployment (call it \texttt{v2}) instead of mutating the existing one, so applications can migrate at their own pace.

The release loop for a new version follows from there. Promote into Staging. Run the golden-set evaluator that production already uses to gate releases. Compare against the current production version on the same set so the comparison is apples-to-apples. Shadow-traffic or canary in production, ramping from one percent to ten to fifty to a hundred. Watch the drift and fairness evaluators in watsonx.governance. If a regression is detected, rollback is one API call, and you investigate before retrying.

\subsection{Model Gateway --- one URL, many backends}

\begin{figure}[h]
\centering
\includegraphics[width=0.95\textwidth]{assets/figures/fig22-model-gateway}
\caption{One Model Gateway in front of multiple backing models. Authentication, rate-limiting, fallback, and observability live in the gateway rather than in every client.}
\label{fig:model-gateway}
\end{figure}

The \product{Model Gateway} is the fronting service that lets multiple models, deployments, and providers sit behind a single API surface. Applications call the gateway; the gateway routes to the right backing model based on rules, headers, A/B configuration, or feature flags. Without it, every application has to know N URLs, N keys, and N retry policies; with it, the policy lives in one place and changes once.

The platform wins are concrete. Multi-model routing (a small cheap model for easy queries, a large flagship for hard ones). Provider abstraction (IBM Granite plus open-source plus a third-party model behind one URL). Centralised authentication, rate-limiting, quota enforcement, and audit logging. A/B and canary at the gateway layer rather than in every client. The Acme Assistant's eventual production architecture had three backing models behind the gateway --- Granite for routine policy answers, Llama-3.3-70B as the fallback when Guardian flagged the Granite answer as low-confidence, and the LoRA-tuned email drafter --- and exactly one URL that the application knew.

\section{SaaS, Software, corpus, and security (Objective 5.5)}

\subsection{SaaS versus Software}

watsonx.ai is offered in two consumption models, and the choice is driven almost entirely by the data, not by features. The \emph{SaaS} (managed) form is IBM-hosted on IBM Cloud. You bring data and prompts; IBM operates the infrastructure. Fastest to start, comes with the standard certifications (SOC~2, ISO~27001), and is the right answer when time-to-value matters and the organisation does not need an in-house ops team. The \emph{Software} form (\emph{Cloud Pak for Data}) deploys watsonx.ai into your own environment --- on-premises, on AWS, Azure, or GCP, or air-gapped. You own operations, and in exchange you get full control over data residency and certifications you cannot satisfy on a managed multi-tenant cloud.

\examtip{If a scenario emphasises \emph{data residency}, \emph{air-gap}, or \emph{sovereign cloud}, the answer is \textbf{Software / Cloud Pak for Data}. If it emphasises \emph{time-to-value} or \emph{no in-house infrastructure team}, the answer is \textbf{SaaS}.}

\subsection{What lives in the Deployment Space, by pattern}

The architectural shape of a deployment changes with the pattern it serves. A RAG deployment carries a prompt template, a retriever configuration, and a generator model; the corpus index itself lives outside the space, in watsonx Discovery, watsonx.data, Elasticsearch, or Milvus. A summarisation deployment carries a prompt template with parameters tuned for length and faithfulness, plus optionally a tuned summariser model. A Q\&A deployment carries a prompt template wired to either parametric Q\&A (closed-book) or to a RAG retriever (open-book).

\subsection{The corpus is part of the deployment}

For RAG and Q\&A, the corpus is not background infrastructure --- it is part of the deployment story. A stale corpus produces a stale answer, with all the citations correctly pointing at last-month's policy. watsonx ships \product{AI Pipelines} as the scheduler that runs ingest, chunk, embed, and index jobs on a cadence (hourly, daily) or on event (object-storage upload). Managing the corpus means adding new sources, removing decommissioned ones, and re-embedding when the embedder version changes. Adjusting the chunking strategy --- moving from 1024-token chunks to 512, for example --- triggers a full re-index of the entire corpus, which is not cheap and needs to be budgeted for.

At Acme, the Monday-morning policy refresh ran as an AI Pipelines job that ingested the diff from compliance's Confluence space, chunked, embedded with slate-125, and re-indexed into watsonx Discovery before the call centre opened on Monday morning. The first time it failed (a corrupt PDF from compliance), the pipeline retried twice, alerted, and the Acme engineer rolled the index back to Sunday's version while compliance re-issued the document.

\subsection{Endpoint security and stability}

A production endpoint needs five things at once. \emph{IAM}: a per-application service ID with a scoped API key, rotated on a cadence. \emph{Network}: prefer private endpoints over public, and deny internet egress where possible. \emph{Rate limits and quotas}: protect against bursts and against runaway loops in calling code. \emph{Input and output guards}: Granite Guardian wrapped around every call for PII, HAP, jailbreaks, and groundedness. \emph{Stability}: multi-zone deployments, health checks, autoscaling, and graceful degradation to a smaller backup model when the primary is degraded. Each of these is the kind of detail the exam will hide in one answer choice; recognising the combination is what makes the right answer obvious.

\subsection{From Prompt Lab to production code}

Prompt Lab can emit ready-to-use client code (Python, curl, or JavaScript) for the current prompt and parameters --- handy for prototyping, less appropriate as a long-term path. In production the better pattern is to call the deployed prompt template via the SDK and let the variable interface stay stable while the prompt itself evolves behind it. The Acme Assistant started with Prompt Lab snippets pasted directly into the microservice; by the time it hit production every call routed through a deployed prompt template, and the microservice never saw the prompt text at all.

\section{A worked example: online and batch}

The end-to-end deployment call has two flavours --- online for interactive users, batch for back-office scoring. The Acme Assistant uses both: online for live agent conversations, batch for the overnight classification of the prior day's tickets.

\begin{examplebox}[title=\textbf{Promote a LoRA-tuned model and create an online deployment}]
\begin{lstlisting}[style=pythonstyle]
# 1) Promote the model asset from the project to a space
promoted = client.repository.promote_asset(
    asset_id        = TUNED_MODEL_ASSET_ID,    # from Tuning Studio
    target_space_id = SPACE_ID_PROD)

# 2) Create a new deployment
meta = {
    client.deployments.ConfigurationMetaNames.NAME: "ticket-classifier-prod",
    client.deployments.ConfigurationMetaNames.DESCRIPTION:
        "LoRA-tuned Granite 3.2 8B, classifier for support tickets.",
    client.deployments.ConfigurationMetaNames.ONLINE: {},
    client.deployments.ConfigurationMetaNames.HARDWARE_SPEC: {
        "name": "L4-1"     # one L4 GPU; pick a profile matching your model size
    },
}
deployment = client.deployments.create(
    artifact_uid = promoted["metadata"]["asset_id"],
    meta_props   = meta)

print("Online URL:", client.deployments.get_href(deployment))
\end{lstlisting}
\end{examplebox}

\begin{examplebox}[title=\textbf{Batch deployment for back-office scoring}]
\begin{lstlisting}[style=pythonstyle]
batch_meta = {
    client.deployments.ConfigurationMetaNames.NAME: "ticket-classifier-batch",
    client.deployments.ConfigurationMetaNames.BATCH: {},
    client.deployments.ConfigurationMetaNames.HARDWARE_SPEC: {"name": "L4-1"},
}
batch_dep = client.deployments.create(
    artifact_uid = promoted["metadata"]["asset_id"],
    meta_props   = batch_meta)

job = client.deployments.create_job(
    deployment_id = client.deployments.get_id(batch_dep),
    meta_props={
        client.deployments.ScoringMetaNames.INPUT_DATA_REFERENCES: [{
            "type": "data_asset",
            "location": {"href": INCOMING_CSV_HREF}
        }],
        client.deployments.ScoringMetaNames.OUTPUT_DATA_REFERENCE: {
            "type": "data_asset",
            "location": {"href": OUTPUT_CSV_HREF}
        },
    })
\end{lstlisting}
\end{examplebox}

\section{Monitoring --- the four pillars}

\begin{figure}[h]
\centering
\includegraphics[width=0.98\textwidth]{assets/figures/fig21-monitoring-pillars}
\caption{The four monitoring pillars and the watsonx tool that owns each. Every ``monitoring'' exam question maps to one of these columns.}
\label{fig:monitoring-pillars}
\end{figure}

A deployed model in production is monitored along four dimensions, and every monitoring question on the exam maps to one of them. \emph{Latency} measures p50, p95, and p99 per endpoint --- the SLA the application owner cares about. \emph{Cost} measures tokens multiplied by rate per tenant --- the line that finance reads. \emph{Drift} compares the live input and output distributions against the baseline captured at deploy time --- the canary in the coal mine for model degradation. \emph{Quality} uses human sampling, LLM-as-judge, and the live evaluators to measure whether the model is still answering correctly.

All four flow into watsonx.governance dashboards. The Acme team configured alerts on drift score above a threshold and on a quality regression greater than two percent, which is conservative for a regulated bank but appropriate to the use case. The exam will sometimes test that you know each pillar has a different owner --- latency belongs to MLOps, cost to finance, drift and quality to the governance lead.

\begin{takeawaybox}
Deployment on watsonx is a discipline, not a button. Projects build; Deployment Spaces serve, and every promotion creates an immutable version that governance can attach to. Applications call deployment endpoints by ID and pass only variables --- never raw prompts and never a model identifier. A custom (tuned, imported) model deploys the same way as a foundation-model prompt template, just with stricter preconditions. The Model Gateway gives the application one URL no matter how many backends you fan out to. Pick SaaS or Software (Cloud Pak for Data) by data residency, not by features. The corpus, for RAG and Q\&A patterns, is part of the deployment story; AI Pipelines is how you keep it fresh. Monitor along four pillars --- latency, cost, drift, quality --- with watsonx.governance as the single dashboard.
\end{takeawaybox}
