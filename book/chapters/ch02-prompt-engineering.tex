\chapter{Prompt Engineering}
\label{ch:prompting}

\epigraph{``Most production AI is prompt engineering. The rest is plumbing.''}{}

The first prompt in an enterprise project is usually too trusting. It assumes the model understands the business, knows the boundaries, remembers the policy, follows the tone guide, refuses unsafe requests, and produces the right format every time. It does none of those things by default. The first version of a typical governed assistant is a single sentence --- something like \emph{``You are a helpful assistant. Answer the user.''} --- and within minutes of testing the system has confidently invented a product, cited a regulation that may or may not exist, or answered in the wrong tone for the brand. A morning of notes from that first test session, all variations on the same theme, is what this chapter turns into a discipline.

Prompt engineering is the discipline of turning a general-purpose model into a controlled interface for a specific task. The exam treats it as Section~2 --- sixteen percent of the test --- and assumes you have already read Chapter~\ref{ch:foundations}, because model selection drives almost every prompt-engineering choice that follows. In this chapter, the prompt becomes an engineering artefact: versioned, parameterised, tested, evaluated, and guarded against hallucination, prompt injection, PII exposure, and unsafe output. By the last page, you will be able to take a prototype prompt and harden it into something the platform team is willing to ship.

\section{Zero-shot, few-shot, and asking the model to think}

\centering

\caption{Direct prompting (left) jumps to a confident wrong answer. Chain-of-thought (right) surfaces verifiable intermediate steps.}

\subsection{Zero-shot}

The cheapest prompt is the one with no examples. You describe the task and the model figures it out from its training. The intent classifier inside a governed assistant typically starts this way --- one sentence describing the task, one ticket, one label out. For a task the model has seen a thousand times in its training data (sentiment, simple summarisation, common-language translation), zero-shot is enough and you should not pay the context-window cost of examples.

\begin{examplebox}[title=\textbf{Zero-shot classification}]
\begin{lstlisting}[style=pythonstyle]
prompt = "Classify the sentiment of the review as POSITIVE, NEGATIVE, or NEUTRAL.\n"\
         "Review: 'Battery died after 30 minutes.'\n"\
         "Sentiment:"
\end{lstlisting}
\end{examplebox}

\subsection{Few-shot}

The moment zero-shot output starts drifting --- in tone, in label vocabulary, in JSON formatting --- the cheapest fix is two to five worked examples placed before the new input. The model imitates the format of what it just saw. Two consistent examples beat ten noisy ones; order them simple-to-complex.

\begin{examplebox}[title=\textbf{Few-shot classification with explicit labels}]
\begin{lstlisting}[style=pythonstyle]
prompt = """Classify each review as POSITIVE, NEGATIVE, or NEUTRAL.

Review: "Amazing camera, exceeded expectations."
Sentiment: POSITIVE

Review: "Battery dies in 30 minutes."
Sentiment: NEGATIVE

Review: "Works as described."
Sentiment: NEUTRAL

Review: "Setup was confusing but it works."
Sentiment:"""
\end{lstlisting}
\end{examplebox}

\subsection{Chain-of-thought}

The last lever in this section is the simplest and the most under-used. Adding the phrase \emph{``Think step-by-step''} before the answer prompt forces the model to write out its reasoning before its conclusion. On multi-step problems --- contract-penalty calculations, multi-policy reasoning, anything with arithmetic --- accuracy jumps and errors become inspectable. The cost is more tokens out. The catch is that a regulator may not want the reasoning trace surfaced to end users, which is why a chain-of-thought system prompt is usually paired with a parser that returns only the final answer to the application while the trace is captured in the audit log.

\examtip{If a scenario shows two to five input-output pairs in the prompt, it is few-shot. If it shows only the task description, it is zero-shot. If it shows \emph{``let's think step by step''} or a numbered reasoning trace, it is chain-of-thought. When the exam phrasing is \emph{``cheapest correct approach''} for a task the model already knows well, the answer is zero-shot.}

\section{Designing a prompt that survives production}

\begin{figure}[h]
\centering
\includegraphics[width=0.95\textwidth]{assets/figures/fig08-prompt-anatomy}
\caption{The five regions of a production prompt. Most exam scenarios test whether you put the right thing in the right region.}
\label{fig:prompt-anatomy}
\end{figure}

A production prompt has five regions, in this order: a \emph{role/system} block that establishes the persona and refusal rules; a \emph{context} block that grounds the model (in RAG, this is where the retrieved chunks land); an \emph{examples} block when you want few-shot; a \emph{task} block with the actual question; and an \emph{output-schema} block that locks the response format. Treat them as separate concerns. When an enterprise assistant misbehaves in testing, ninety percent of the time it is because two of these regions were collapsed into one and the model could not tell instruction from content.

Chat-style and freeform prompts are not the same shape on the wire. A chat-capable model takes a structured list of \texttt{(role, content)} messages and honours the \texttt{system} role separately from user input --- that separation is itself a defence against prompt injection. A freeform call sends one big string. Use the chat shape for anything customer-facing.

\begin{examplebox}[title=\textbf{Chat-style call with system / user roles}]
\begin{lstlisting}[style=pythonstyle]
from ibm_watsonx_ai.foundation_models import ModelInference

model = ModelInference(model_id="ibm/granite-3-2-8b-instruct",
                       api_client=client,
                       params={"decoding_method": "greedy",
                               "max_new_tokens": 250,
                               "temperature": 0.3})

messages = [
    {"role": "system",
     "content": "You are a polite, concise IT helpdesk agent."},
    {"role": "user",
     "content": "My laptop won't connect to the office Wi-Fi."},
]
print(model.chat(messages=messages))
\end{lstlisting}
\end{examplebox}

For translation, two extra habits matter. State both the source and target language explicitly --- never assume the model will detect them correctly. Tell the model to preserve formatting (Markdown, JSON keys, line breaks) if your downstream pipeline needs it. For long content, chunk by paragraph, translate each, and re-join; that avoids silent context-window truncation when the source document is a few thousand tokens long.

\begin{examplebox}[title=\textbf{Translation prompt with formatting rules}]
\begin{lstlisting}[style=pythonstyle]
prompt = """Translate the English text into French.
Preserve technical terms and keep the same paragraph structure.

English: "Restart the router and reconnect to the corporate SSID."
French:"""
\end{lstlisting}
\end{examplebox}

\subsection{Three frameworks, one habit}

The exam mentions three prompt-design frameworks --- \textbf{CRISPE}, \textbf{APE}, \textbf{CARE} --- and you should be able to name them. CRISPE prompts you to write a \emph{Capacity, Role, Insight, Statement, Personality, Experiment} block; APE structures \emph{Action, Purpose, Expectation}; CARE asks for \emph{Context, Action, Result, Example}. In practice every team I have worked with picks one and uses it as a checklist. They are all variations on the same idea: explicit role, explicit task, explicit output. Read the prompt anatomy figure once; you have absorbed the substance of all three.

\section{Prompt templates: code, not strings}

A prompt that lives as a Python f-string in your microservice is technical debt. The watsonx \term{prompt template} is a named, versioned, governed asset with input variables, a model binding, parameters, and (optionally) few-shot examples. You promote it through deployment spaces the same way you promote a model. You attach evaluation results to it. You can swap the underlying model without touching application code. Treat it like source code, because it is.

\begin{examplebox}[title=\textbf{Create and store a prompt template}]
\begin{lstlisting}[style=pythonstyle]
from ibm_watsonx_ai.foundation_models.prompts import (
    PromptTemplate, PromptTemplateManager
)

ptm = PromptTemplateManager(credentials=creds, project_id=PROJECT_ID)

template = PromptTemplate(
    name="ticket-classifier-v1",
    description="Classify support tickets into BILLING|TECH|OTHER.",
    input_variables=["ticket_text"],
    input_text=("Categorise the support ticket.\n"
                "Ticket: {ticket_text}\nCategory:"),
    examples=[
        ["My charge is wrong",   "BILLING"],
        ["The app keeps crashing", "TECH"],
    ],
    model_id="ibm/granite-3-2-8b-instruct",
    model_params={"decoding_method": "greedy",
                  "max_new_tokens": 6,
                  "stop_sequences": ["\n"]},
)
stored = ptm.store_prompt(template)
print("Asset ID:", stored.prompt_id)
\end{lstlisting}
\end{examplebox}

\subsection{Evaluating a template before you promote it}

Before a template moves out of the project, evaluate it. For classification, use accuracy or F1 against a labelled held-out set; for summarisation or translation, use ROUGE or BLEU; for extraction, use exact-match on the field values. For anything open-ended where deterministic scoring is hard, use an LLM-as-judge with a written rubric --- a stronger model grades the outputs and produces a score. The evaluation results attach to the prompt asset itself in watsonx.governance, which is what makes the audit trail real.

The promotion path is the same one Chapter~\ref{ch:deployment} covers in depth: Project (where you build) $\rightarrow$ Dev Space (end-to-end test) $\rightarrow$ Staging (governance factsheet attaches here) $\rightarrow$ Production (canary, then ramp).

\begin{examplebox}[title=\textbf{Promote and deploy the template}]
\begin{lstlisting}[style=pythonstyle]
# Promote the asset into a deployment space
new_asset = client.repository.promote_asset(asset_id=stored.prompt_id,
                                            target_space_id=SPACE_ID)

# Create an online deployment
meta = {
    client.deployments.ConfigurationMetaNames.NAME: "ticket-classifier-prod",
    client.deployments.ConfigurationMetaNames.ONLINE: {},
    client.deployments.ConfigurationMetaNames.BASE_MODEL_ID: "ibm/granite-3-2-8b-instruct",
}
deployment = client.deployments.create(artifact_uid=new_asset["metadata"]["asset_id"],
                                       meta_props=meta)
print(client.deployments.get_id(deployment))
\end{lstlisting}
\end{examplebox}

\examtip{When an exam question describes \emph{a versioned, governed, audit-friendly prompt}, the answer is a prompt template in a deployment space, not a free-form prompt in an SDK call.}

\section{Decoding parameters: how the next token is picked}

\begin{figure}[h]
\centering
\includegraphics[width=0.95\textwidth]{assets/figures/fig09-decoding-parameters}
\caption{How temperature, top-$k$, and top-$p$ reshape the next-token distribution. Left: greedy / low temperature. Right: higher temperature with both pruning strategies applied.}
\label{fig:decoding-parameters}
\end{figure}

After the model produces a probability distribution over its vocabulary, something has to actually pick the next token. That choice is decoding, and watsonx splits it into two top-level methods. \emph{Greedy} always picks the highest-probability token --- deterministic, the right choice for factual extraction, classification, anything where you want the same output for the same input. \emph{Sampling} picks from the distribution, optionally shaped by temperature, top-$k$, and top-$p$ --- the right choice for creative or diverse output.

\emph{Temperature} reshapes the distribution before sampling. Higher temperature flattens it (more diverse, more creative, more drift); lower temperature peaks it (more deterministic, more conservative). At the extreme, temperature approaching zero is equivalent to greedy. Sensible range: zero to one for production prompts, up to two for brainstorming, never above.

\emph{Top-$k$} and \emph{top-$p$} prune the distribution before sampling. Top-$k$ keeps only the $k$ highest-probability tokens; small $k$ (10--40) restricts wild outputs, large $k$ lets rare-but-still-likely tokens through. Top-$p$ (nucleus sampling) keeps the smallest set of tokens whose cumulative probability exceeds $p$; $p = 0.9$ is the popular default. Top-$p$ is more adaptive: in confident contexts the nucleus is tiny, in ambiguous contexts it widens. Most teams I've worked with use top-$p$ alone or both together, and almost nobody uses top-$k$ on its own anymore.

\emph{Repetition penalty} (typical range 1.0 to 1.2) lowers the probability of tokens that have already appeared in the output --- the fix when the model produces \emph{``thank you thank you thank you''}. Push it above 1.5 and the model starts paraphrasing itself or going off-topic.

\emph{Random seed} matters only when sampling. Same prompt, same seed, same parameters $\Rightarrow$ same output. Useful for reproducible tests and demos; not relevant for greedy decoding.

\emph{Stopping criteria} answer the question \emph{when does the model stop generating?} The generation halts as soon as any of these triggers: \texttt{max\_new\_tokens} (a hard ceiling on output length), \texttt{stop\_sequences} (a list of strings the model is told to stop on), \texttt{time\_limit} (milliseconds per call --- the SLA enforcer), the model's own end-of-sequence token (\texttt{</s>}, \texttt{<|endoftext|>}, depending on family). \texttt{min\_new\_tokens} works the other way --- it forces the model past a minimum length before any stop logic fires, which prevents pathological one-word answers like \emph{``OK''} on summarisation tasks.

\begin{examplebox}[title=\textbf{A complete parameter dictionary}]
\begin{lstlisting}[style=pythonstyle]
params = {
    "decoding_method":      "sample",   # or "greedy"
    "temperature":          0.7,
    "top_k":                50,
    "top_p":                0.9,
    "random_seed":          42,
    "repetition_penalty":   1.1,
    "min_new_tokens":       10,
    "max_new_tokens":       250,
    "stop_sequences":       ["\n###", "</answer>"],
    "time_limit":           20000,      # milliseconds
}
\end{lstlisting}
\end{examplebox}

\begin{examplebox}[title=\textbf{Symptom-to-parameter cheat sheet}]
\begin{lstlisting}[style=pythonstyle]
# Output is too random / off-topic
params["temperature"]  = 0.2; params["top_p"] = 0.5; params["decoding_method"] = "greedy"

# Output is too short
params["min_new_tokens"] = 50

# Output loops / repeats phrases
params["repetition_penalty"] = 1.15

# Output runs forever
params["max_new_tokens"]  = 300
params["stop_sequences"]  = ["\n\n", "###"]

# Need reproducibility for tests
params["decoding_method"] = "sample"; params["random_seed"] = 42
\end{lstlisting}
\end{examplebox}

\section{Prompt variables and Prompt Lab}

A prompt \emph{variable} is a placeholder in the template (\texttt{\{ticket\_text\}}, \texttt{\{language\}}, \texttt{\{tone\}}) that the application fills at call time. Variables are not cosmetic. They give you reusability (one template, many calls), safety (the SDK escapes and quotes them rather than your app concatenating strings, which shrinks the prompt-injection surface), governance (a single asset to audit instead of $N$ free-form strings), and the ability to A/B test by swapping the model behind the same variable contract. The rule is simple: anything that changes per request is a variable; anything that stays the same is static prompt text.

\subsection{Prompt Lab, in three modes}

Prompt Lab is the no-code prompt builder inside watsonx.ai, and the exam expects you to recognise its three editing modes by what they are best at. \emph{Chat} mode gives you a system block and a multi-turn conversation pane --- the right mode for chatbots. \emph{Structured} mode separates instruction, examples, and input into fields --- the right mode for few-shot tasks with managed variables. \emph{Freeform} mode is one big text box with no UI guardrails --- the right mode for power users with custom prompt formats.

\section{The four risk families every prompt has to survive}

The same four risk families appear in Chapter~\ref{ch:foundations} under the security section. Here they show up again because the prompt is where most of them are mitigated. The production prompt of any governed assistant evolves through all four before it ships.

\emph{Hallucination} is the model producing a confident, plausible answer that is factually wrong. Underlying causes: it is a probabilistic next-token predictor; the training data was missing or contradictory; the prompt is out-of-distribution; the temperature was too high. The mitigation stack is layered: ground the model with RAG (Chapter~\ref{ch:rag}) so it has source text to lean on, lower the temperature, use greedy decoding for factual tasks, add an explicit refusal instruction (\emph{``If the answer is not in the context, say `I do not know'.''}), and for high-stakes outputs add a verification step --- a second LLM call as a fact-checker, or Granite Guardian's groundedness check on the output.

\emph{Personally identifiable information} (PII) is names, emails, phone numbers, government IDs, account numbers, addresses, IP addresses, dates of birth --- anything that identifies a person. Mitigation runs at three layers: a PII filter on the input (regex or NER pre-mask, replace masked tokens after generation), Granite Guardian on the output to refuse leaks, and never logging raw prompts (redact at the logging layer). Data residency matters here too --- pick a watsonx region that matches the regulatory regime of your data.

\emph{Hate, abuse, profanity} (HAP) is its own category because regulators treat it that way. The watsonx HAP classifier (now folded into the Granite Guardian family) scores text on hate, abuse, and profanity and is applied on both input and output. There is no clever prompt-level fix here; you wrap the call with Guardian and trust the classifier.

\emph{Bias} comes from imbalanced training data, biased annotators, or biased proxy features in the input. The mitigation runs at three levels too: neutralising instructions in the system prompt; a balanced few-shot set if you are using one; and demographic-parity / equalised-odds tests in watsonx.governance, with the residual bias documented in the model card. Bias is never fully solved, and the exam may test that you know this --- the right answer is usually \emph{``mitigate at multiple layers and document''}, not \emph{``solve at one layer.''}

\examtip{Memorise the four risk filter families --- PII, HAP, bias, and hallucination/groundedness. They map directly to Granite Guardian categories and to watsonx.governance evaluators. The exam loves questions that pair a symptom (\emph{``the model invented a citation''}, \emph{``the model produced a slur''}) to one of these four.}

\subsection{A reference production prompt}

A first production prompt for a governed support assistant --- after the morning of failed test queries that opens this chapter --- typically ends up looking like the block below. Notice the five regions from Figure~\ref{fig:prompt-anatomy} all present and labelled, the explicit refusal instruction, and the locked output schema.

\begin{examplebox}[title=\textbf{Reference production prompt skeleton}]
\begin{lstlisting}[style=pythonstyle]
SYSTEM = """You are a senior support agent at the enterprise.
Answer ONLY from the policy excerpts in <context>.
If the answer is not in the context, reply:
  'I do not know -- please contact a human agent.'
Tone: formal, concise. Output strict JSON:
  {"answer": str, "policy_id": str | null, "confidence": "high"|"low"}"""

def build_prompt(question: str, chunks: list[str]) -> list[dict]:
    context = "\n\n".join(f"[{i+1}] {c}" for i, c in enumerate(chunks))
    return [
        {"role": "system", "content": SYSTEM},
        {"role": "user",
         "content": f"<context>\n{context}\n</context>\n\nQuestion: {question}"},
    ]

params = {
    "decoding_method":    "greedy",
    "max_new_tokens":     350,
    "temperature":        0.2,
    "stop_sequences":     ["}\n", "</answer>"],
    "time_limit":         20000,
}
\end{lstlisting}
\end{examplebox}

\begin{takeawaybox}
Prompt engineering on the exam has three layers and you should be able to move between them in any question. The \emph{structural} layer asks whether the right thing is in the right region of the prompt --- role, context, examples, task, schema. The \emph{decoding} layer asks how the next token is chosen --- greedy for facts, sampling with temperature plus top-$p$ for creativity, and the right stopping criteria so the call has a finite cost. The \emph{operational} layer asks how the prompt is shipped --- a versioned prompt template promoted through deployment spaces, evaluated against a held-out set, attached to a governance factsheet, and wrapped with Granite Guardian on input and output for the four risk families.
\end{takeawaybox}

\section{Practice questions}

\begin{practicebox}[title=\textbf{Q2.1 --- Decoding}]
You need consistent, factual, reproducible output for an extraction task. Best decoding method?

\textbf{A.} \texttt{sample} with temperature=1.0\\
\textbf{B.} \texttt{greedy}\\
\textbf{C.} \texttt{sample} with top\_p=0.99\\
\textbf{D.} beam search with width 10

\vspace{0.2cm}\textit{Answer: B. Greedy is deterministic by definition.}
\end{practicebox}

\begin{practicebox}[title=\textbf{Q2.2 --- Parameter pick}]
The model keeps repeating the same phrase. \emph{Single} change with the biggest payoff?

\textbf{A.} Lower temperature to 0\hspace{1em}\textbf{B.} Increase \texttt{max\_new\_tokens}\\
\textbf{C.} Raise \texttt{repetition\_penalty} to $\sim$1.15\hspace{1em}\textbf{D.} Switch to a smaller model

\vspace{0.2cm}\textit{Answer: C.}
\end{practicebox}

\begin{practicebox}[title=\textbf{Q2.3 --- Stop criteria}]
You ship a chatbot and need to cap each call at 20 seconds and 300 generated tokens. Which params?

\textbf{A.} \texttt{stop\_sequences=["20s"]}\\
\textbf{B.} \texttt{max\_new\_tokens=300, time\_limit=20000}\\
\textbf{C.} \texttt{min\_new\_tokens=300, temperature=20}\\
\textbf{D.} \texttt{repetition\_penalty=20}

\vspace{0.2cm}\textit{Answer: B.}
\end{practicebox}

\begin{practicebox}[title=\textbf{Q2.4 --- Prompt template}]
You want versioning, governance, and the ability to swap models without changing app code. What do you ship?

\textbf{A.} Hard-coded f-string in your microservice\\
\textbf{B.} \texttt{PromptTemplate} stored in a watsonx project and promoted to a deployment space\\
\textbf{C.} A YAML file in your repo\\
\textbf{D.} A Jinja template on your web server

\vspace{0.2cm}\textit{Answer: B.}
\end{practicebox}

\begin{practicebox}[title=\textbf{Q2.5 --- Prompt Lab mode}]
A power user wants to hand-craft an unusual prompt with custom delimiters and no UI guardrails. Which mode?

\textbf{A.} Chat\hspace{1em}\textbf{B.} Structured\hspace{1em}\textbf{C.} Freeform\hspace{1em}\textbf{D.} JSON-only

\vspace{0.2cm}\textit{Answer: C.}
\end{practicebox}

\begin{practicebox}[title=\textbf{Q2.6 --- Risks}]
A model outputs a believable but invented case-law citation. Category?

\textbf{A.} Prompt injection\hspace{1em}\textbf{B.} Hallucination\hspace{1em}\textbf{C.} PII leak\hspace{1em}\textbf{D.} Bias

\vspace{0.2cm}\textit{Answer: B. Mitigate with RAG, low temperature, Granite Guardian groundedness check, and a citation requirement in the system prompt.}
\end{practicebox}

\begin{practicebox}[title=\textbf{Q2.7 --- HAP / PII}]
Which two filter families should you wrap around \emph{every} consumer-facing watsonx.ai endpoint?

\textbf{A.} Authentication + rate limit\\
\textbf{B.} PII + HAP via Granite Guardian\\
\textbf{C.} top\_k + top\_p\\
\textbf{D.} repetition\_penalty + min\_new\_tokens

\vspace{0.2cm}\textit{Answer: B.}
\end{practicebox}
