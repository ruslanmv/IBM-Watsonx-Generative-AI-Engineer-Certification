\chapter{watsonx --- Integration and Model Orchestration}
\label{ch:integration}

\epigraph{``A model in isolation is a demo. A model wired into your business is a product.''}{}

The model is rarely the system. In production, the model is one component inside a larger workflow: user interface, identity, retrieval, tools, business systems, policy checks, logging, monitoring, and human escalation. A governed support assistant typically has to talk to three different systems on every interaction --- a chat panel embedded in the agent console, a CRM that holds account state, a ticketing system that holds open cases --- on top of the retrieval and generation layers the earlier chapters built. Integration is the work of making those parts behave as one product.

Section~6 is eight percent of the exam --- the smallest section by weight. It is also the section whose topics show up scattered across every other section, because almost every realistic exam scenario implies an integration somewhere. The objectives walk a clean line: pair watsonx.ai with the rest of the IBM cloud (6.1), call models through the SDK and REST with proper error handling (6.2), use webhooks for event-driven flows (6.3), build chains, tools, and agents with LangChain on watsonx (6.4), and recognise the Model Context Protocol when it shows up (6.5). The chapter closes with the production architecture most enterprise watsonx deployments converge on.

\section{Pairing watsonx.ai with the rest of the stack (Objective 6.1)}

watsonx.ai rarely lives alone. The cluster of services it most often pairs with is small enough to list in one paragraph, and the exam will test whether you recognise the pairings on sight. \product{Watson Assistant} is the conversational front-end --- it lives on the web, the phone, Teams, or WhatsApp; you plug watsonx.ai behind it for generative replies and hand off to live agents on escalation. \product{Watson Discovery} (the legacy product) and \product{watsonx Discovery} (the current one) are the document-understanding and hybrid-retrieval services, the primary RAG retrievers for new builds. \product{watsonx.data} is the governed lakehouse, the source of structured data for prompts and tuning. \product{watsonx.governance} is the audit layer --- factsheets, drift, fairness, model risk management. IBM Cloud Functions and Code Engine are the serverless wrappers around the API. Event Streams (managed Kafka) handles the asynchronous workflows.

The integration points themselves are straightforward. Everything in watsonx is reachable via REST endpoints under \texttt{\{region\}.ml.cloud.ibm.com}; the official SDKs are \texttt{ibm-watsonx-ai} for Python and \texttt{@ibm-cloud/watsonx-ai} for JavaScript and TypeScript, with LangChain bindings shipping in \texttt{langchain-ibm}. Watson Assistant can call watsonx via webhook for dynamic responses, and watsonx.data and Cloud Pak for Data ship pre-built connectors for most enterprise data sources.

Configuration happens through either the UI or the API. The UI path is the friendly one: in the watsonx project, add the integration partner (Watson Assistant, for example) via the Integrations tab, sign in, and map credentials. The API path is the production one: store the partner's API key in a secrets manager (IBM Cloud Secrets Manager or HashiCorp Vault), inject it at runtime, configure the partner's webhook to call the watsonx deployment URL, and never write the key into application configuration. Either way, every deployment automatically registers with the governance catalogue, where you attach evaluators (drift, fairness, quality) and produce a factsheet per model version. The factsheet is the deliverable an auditor will ask for.

\subsection{The shape of an external call}

The cleanest demonstration of the API surface is a raw curl, because it works in every language and exposes exactly what the SDK hides. A typical external integration test --- the one a platform team runs every release to verify the deployment URL is reachable through the corporate proxy --- looks like the block below.

\begin{examplebox}[title=\textbf{Authenticated REST call from any language (curl)}]
\begin{lstlisting}
# Get an IAM token
TOKEN=$(curl -s -X POST https://iam.cloud.ibm.com/identity/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=urn:ibm:params:oauth:grant-type:apikey&apikey=$WATSONX_APIKEY" \
  | jq -r .access_token)

# Call a chat deployment
curl -X POST "https://us-south.ml.cloud.ibm.com/ml/v1/deployments/$DEPLOYMENT_ID/text/chat?version=2024-08-01" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
        "messages": [
          {"role":"system","content":"You are a helpful assistant."},
          {"role":"user","content":"Hello!"}
        ]
      }'
\end{lstlisting}
\end{examplebox}

\subsection{API keys are not optional infrastructure}

Key management is one of those topics where the exam answer is also the production answer. Create a service ID per application --- never per developer, never one shared across services. Generate an API key scoped to that service ID with the minimum role for the work (usually \emph{Reader} on the watsonx service, \emph{Writer} only if the application trains or deploys). Store the key in IBM Cloud Secrets Manager and inject at runtime; never commit a key to git, and reject any pull request that contains one. Rotate on a schedule and on personnel changes. None of this is surprising; the exam tests that you know it cold.

\subsection{Errors the SDK will raise, and what to do about each}

The Python SDK raises \texttt{WMLClientError} for failures; via raw REST you get HTTP 4xx or 5xx. Six status codes appear often enough to memorise. A \texttt{400} is a bad request --- malformed JSON, an invalid \texttt{model\_id}; fix the request, do not retry. A \texttt{401} is unauthorised --- the token expired or the API key is wrong; refresh and retry. A \texttt{403} is forbidden --- the service ID does not have the role for this operation; an IAM change is needed. A \texttt{404} is not found --- wrong project, space, or deployment ID. A \texttt{429} is rate-limited --- back off with exponential jitter and retry. The \texttt{500}, \texttt{502}, and \texttt{503} family are transient server or gateway errors; retry up to three times. A \texttt{504} is a timeout: the response exceeded the time budget, so either trim the prompt or raise the \texttt{time\_limit} parameter.

\begin{examplebox}[title=\textbf{Retry-with-backoff helper}]
\begin{lstlisting}[style=pythonstyle]
import time, random
from ibm_watsonx_ai.wml_client_error import WMLClientError

def call_with_retry(fn, *args, max_attempts=4, **kw):
    for attempt in range(1, max_attempts + 1):
        try:
            return fn(*args, **kw)
        except WMLClientError as e:
            transient = any(s in str(e) for s in ("429", "500", "502", "503"))
            if not transient or attempt == max_attempts:
                raise
            sleep_s = 2 ** attempt + random.random()
            time.sleep(sleep_s)
\end{lstlisting}
\end{examplebox}

\section{Orchestrating workflows (Objective 6.2)}

A single LLM call is almost never the answer in production. A typical reply-to-customer-email workflow has seven steps the first time it ships: ingest the customer ticket, extract entities, classify the intent, retrieve the relevant policy chunks, draft a reply, check the reply with Granite Guardian, and either send it or escalate to a human. That sequence is exactly what \emph{orchestration} means in watsonx.

Designing such a workflow follows four habits. Break the task into atomic steps, each one taking a dictionary in and returning a dictionary out. Identify which steps are LLM calls, which are tool calls (a database query, a third-party API), and which are deterministic Python (a regex, a formatter, a checksum). Decide which steps can run in parallel --- entity extraction and intent classification can usually run together --- and which need strict ordering. And define the failure or fallback behaviour for each step before you write the happy path, because the unhappy paths are where production breaks.

The tools you orchestrate with sit at different levels of the stack. \product{LangChain Chains} cover linear or DAG-style flows with the smallest learning curve. \product{LangGraph} models explicit state machines with conditional edges and is the right tool when reproducibility matters and the flow has loops. \product{watsonx AI Pipelines} is the managed scheduler for data and AI jobs --- Spark, notebooks, deployments --- with cron triggers, event triggers, and dependency management. IBM Cloud Code Engine and Functions wrap event-driven steps in serverless. The exam will test that you recognise each by name and pick the right one for the scenario.

Automation, observability, and the monitoring habits that come with workflows are the same ideas applied at the step level. Schedule with AI Pipelines cron, with Watson Assistant message triggers, or with Event Streams. Manage dependencies as a DAG with explicit parents. Set a per-step on-failure policy --- retry, branch, alert, dead-letter queue --- so partial failures do not corrupt downstream state. Branch conditionally on classifier output, retrieved score, or Guardian verdict. Log every step to a unified store (IBM Cloud Logs, CloudWatch, Splunk). Capture cost (tokens), latency, and quality (LLM-judge or human sample) per step. The bottleneck is almost always either re-ranking or the largest model in the chain; parallelise the independent calls, cache the deterministic steps, and downsize models where quality allows.

Agentic RAG, covered in Chapter~\ref{ch:rag}, is one node inside a larger orchestration. The agent itself decides whether and when to retrieve --- the workflow around it decides what to do with the result.

Given any business case, the design walks five questions in order. What are the input channels (web chat, mobile app, email, an internal back-office app)? What are the data sources (CRM, policy repository, billing database, ticketing system)? Where is the trust boundary --- where does data leave your network, and where can it not? Which watsonx components do you bring in (\texttt{watsonx.ai} for generation, \texttt{watsonx Discovery} for retrieval, \texttt{watsonx.data} for the corpus, \texttt{watsonx.governance} for audit)? And which external services need to participate (Watson Assistant as the front-end, IAM, secrets, monitoring, queues)? Answer those five questions and the architecture diagram draws itself.

The reference architecture below is the shape most enterprise watsonx deployments converge on. A web or mobile UI feeds into Watson Assistant, which routes generative replies through a Model Gateway into watsonx.ai. Behind watsonx.ai sits a LangGraph workflow that retrieves from watsonx Discovery (which itself draws structured data from watsonx.data when needed) and runs Granite Guardian on both sides of the model. Lineage and metrics flow into watsonx.governance for the audit trail.

\subsection{Model Context Protocol (Objective 6.5)}

\begin{figure}[h]
\centering
\includegraphics[width=0.98\textwidth]{assets/figures/fig24-mcp-architecture}
\caption{Left: an MCP-enabled agent speaks one protocol (JSON-RPC) to many servers (Postgres, CRM, Slack). Right: without MCP, every integration is custom code in the agent.}
\label{fig:mcp}
\end{figure}

The \term{Model Context Protocol} (MCP) is an open protocol for connecting LLM applications to external tools and data sources in a standard way. An \emph{MCP server} exposes resources (data) and tools (functions); an \emph{MCP client} --- the LLM application, agent, or IDE --- discovers and uses them through a common JSON-RPC interface. The contrast with the older approach is the whole point: instead of writing a bespoke LangChain tool for every integration (one for Postgres, one for Slack, one for the internal CRM, one for the ticketing system), you stand up an MCP server per system once, and any MCP-capable agent can call any of them through the same protocol.

Why the exam cares: MCP is the direction IBM is pushing for plugging external systems into agentic watsonx workflows. You point your agent at the MCP server and it can call any of the tools that server exposes, without per-integration Python code. A common refactor inside a maturing program is to move a CRM integration from a bespoke LangChain tool to an MCP server --- the diff is a removal of a few hundred lines of glue code and the addition of one configuration block.

Three properties are worth remembering. MCP standardises tool discovery and invocation through a vendor-neutral JSON-RPC interface, which means the same agent code works against different servers. It separates the LLM application from each integration's specifics, which means the agent does not need to know whether it is talking to Postgres or Slack. And it plays well with native tool-calling chat models --- Granite 3.x, Llama-3.3, and Mistral all support tool calling, so an MCP-discovered tool can be called by any of them through the same code path.

\section{LangChain on watsonx (Objective 6.4)}

\subsection{The vocabulary}

LangChain is the framework that most teams build watsonx applications on. The exam expects you to recognise its core building blocks by name and use.

\begin{table}[h]
\centering
\small
\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{Concept}        & \textbf{What it is} \\
\midrule
\textbf{LLM}            & A wrapper around a foundation model (\texttt{WatsonxLLM}, \texttt{ChatWatsonx}) \\
\textbf{Prompt template}& A reusable prompt with input variables \\
\textbf{Chain}          & A sequence of steps (prompt $\to$ LLM $\to$ parser, etc.) \\
\textbf{Tool}           & A Python function the agent can call (search, calculator, DB) \\
\textbf{Agent}          & LLM controller that picks the next tool based on the goal \\
\textbf{Memory}         & Short-term (scratchpad) or long-term (vector store) state \\
\textbf{Retriever}      & Wraps a vector store or search index for RAG \\
\textbf{Output parser}  & Coerces LLM text into structured Python or JSON \\
\bottomrule
\end{tabularx}
\caption{LangChain core building blocks.}
\label{tab:langchain-blocks}
\end{table}

The reasons LangChain has won by adoption are mundane. \emph{Modularity}: swap LLM, retriever, or memory without rewriting the pipeline. \emph{Reusability}: chain and tool libraries shared across teams. \emph{Extensibility}: custom tools, custom retrievers, custom output parsers. None of those is exciting; together they are why most watsonx production code in \the\year{} is LangChain on top.

\subsection{From a chain to an agent}

\begin{figure}[h]
\centering
\includegraphics[width=0.98\textwidth]{assets/figures/fig23-langchain-chain}
\caption{Top: a deterministic LangChain pipeline (prompt $\to$ LLM $\to$ parser). Bottom: an agent that loops with tools and memory until it has enough evidence to answer.}
\label{fig:langchain}
\end{figure}

The simplest useful LangChain pipeline is a deterministic chain --- prompt, LLM, parser, piped together with the \texttt{|} operator. That is what a typical intent classifier looks like in week one.

\begin{examplebox}[title=\textbf{LLMChain --- prompt + watsonx LLM + parser}]
\begin{lstlisting}[style=pythonstyle]
from langchain_ibm import WatsonxLLM
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser

llm = WatsonxLLM(model_id="ibm/granite-3-2-8b-instruct",
                 url=URL, apikey=API_KEY, project_id=PROJECT_ID,
                 params={"decoding_method": "greedy",
                         "max_new_tokens": 250, "temperature": 0.2})

prompt = ChatPromptTemplate.from_messages([
    ("system", "Return strict JSON: {\"intent\": ..., \"priority\": ...}"),
    ("user",   "Ticket: {ticket_text}")
])

chain = prompt | llm | JsonOutputParser()
print(chain.invoke({"ticket_text": "My internet is down"}))
\end{lstlisting}
\end{examplebox}

Where the workflow needs to call out to something else --- a database, an internal API, a third-party service --- a chain stops being enough and you reach for a tool. A LangChain tool is just a decorated Python function with a docstring the LLM reads to decide whether to call it.

\begin{examplebox}[title=\textbf{Custom tool --- query an internal CRM}]
\begin{lstlisting}[style=pythonstyle]
from langchain_core.tools import tool
import requests

@tool
def lookup_account(account_id: str) -> dict:
    """Return account name, tier, status from the internal CRM."""
    r = requests.get(f"https://crm.internal/api/accounts/{account_id}",
                     headers={"Authorization": f"Bearer {CRM_TOKEN}"})
    r.raise_for_status()
    return r.json()
\end{lstlisting}
\end{examplebox}

Wire one or more tools to a chat model with native tool-calling support and you have an agent. Granite~3.x, Llama-3.3, and Mistral all support tool calling natively, which is the modern way to build agents and the one the exam tests.

\begin{examplebox}[title=\textbf{Tool-calling agent on watsonx (Granite 3.x supports tool calls natively)}]
\begin{lstlisting}[style=pythonstyle]
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_ibm import ChatWatsonx
from langchain_core.prompts import ChatPromptTemplate

llm = ChatWatsonx(model_id="ibm/granite-3-2-8b-instruct", ...)

tools = [lookup_account, search_kb, send_email]   # custom tools

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a senior support agent. Use the tools provided. "
               "If unsure, escalate."),
    ("human",  "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])

agent     = create_tool_calling_agent(llm, tools, prompt)
executor  = AgentExecutor(agent=agent, tools=tools, verbose=True)

executor.invoke({"input":
    "Account 882 reports an outage --- summarise their status and draft a reply."})
\end{lstlisting}
\end{examplebox}

Memory is the dimension that turns a stateless agent into a multi-turn assistant. \texttt{ConversationBufferMemory} holds the exact transcript and is cheap to implement but expensive in tokens at long horizons. \texttt{ConversationSummaryMemory} maintains a running summary in an LLM and is cheaper at long horizons but loses detail. \texttt{VectorStoreRetrieverMemory} gives semantic recall across past turns when the conversation is too long to fit in context. LangGraph checkpoints persist agent state across sessions, which is what production deployments use when conversations span days.

\subsection{The end-to-end picture}

Most production builds look like this end to end. Watson Assistant catches the user's intent and channel. For generative replies it calls a watsonx Deployment via webhook. Behind the deployment, a LangChain or LangGraph workflow does RAG over watsonx Discovery, possibly calls an MCP-exposed internal API, drafts the answer with Granite, and runs Granite Guardian over both input and output. Lineage and metrics land in watsonx.governance for the audit. Every chapter in this book has been a slice of this same architecture.

\begin{takeawaybox}
Integration on watsonx is the discipline of wiring a model into the rest of an organisation's systems without losing the audit trail. Calls go through the REST API or the SDKs, authenticated by per-application service IDs and scoped API keys. External calls always wrap in retries, backoff, and category-specific error handling. Workflows orchestrate with LangChain Chains for simple flows, LangGraph for state-machine reproducibility, and watsonx AI Pipelines for scheduled data-plus-AI jobs. LangChain's core blocks --- chains, tools, agents, memory, retrievers --- cover the agent layer; the Model Context Protocol covers the tools layer when the same integrations are reused across applications. Granite~3.x, Llama-3.3, and Mistral all support native tool calling, which is the modern way to build agents on watsonx. And the production architecture that all of this converges on is the one in Figure~\ref{fig:reference-architecture}.
\end{takeawaybox}
